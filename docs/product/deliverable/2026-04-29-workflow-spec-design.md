# 项目开发与调试工作流规范

## 概述

本规范定义了适用于所有项目的开发与调试工作流程，持久化于全局 `~/.claude/CLAUDE.md`，确保每次会话自动加载。

## 一、全局规则

以下规则适用于所有项目、所有工作场景：

1. **Karpathy 准则强制**：全局 `~/.claude/CLAUDE.md` 中固定包含 karpathy-guidelines 的 4 条行为准则核心要求（Think Before Coding、Simplicity First、Surgical Changes、Goal-Driven Execution），所有项目自动生效。
2. **禁止自行编码**：在获得用户对开发计划的明确审批之前，不得编写任何实现代码。仅允许探索代码库、阅读文件等研究行为。
3. **功能分支**：所有新功能开发和 debug 修复必须从主干拉取独立分支（`feature/<topic>` 或 `bugfix/<topic>`），完成后合并回主干。
4. **文档语言**：所有项目文档使用中文（zh-CN）。
5. **完成定义**：一个开发/debug 任务视为完成，需同时满足：
   - 代码审查（`/code-review`）通过
   - 构建成功（`bun run build` 或项目对应构建命令）
   - 用户明确确认需求已完成
   - 文档已移动至 `done/` 目录
   - 经提示后按用户选择完成代码风格和安全性审查（或用户跳过）

## 二、文档目录与流转规范

### 目录结构

```
docs/
├── product/          # 产品文档
│   ├── context/      # 用户主动提供的输入、背景文件、原始需求
│   ├── deliverable/  # 确认进入开发流程的文档
│   ├── raw/          # 中间产物、实验性质文档、不确定类型的文档
│   └── tools/        # 过程中产生的可复用工具、流程
├── development/      # 开发文档
│   ├── todo/         # 尚未开始
│   ├── doing/        # 正在开发
│   └── done/         # 已完成
└── debug/            # Debug 文档
    ├── todo/         # 尚未开始
    ├── doing/        # 正在调试
    └── done/         # 已完成
```

### 通用规则

- **命名规则**：`YYYY-MM-DD-<topic>.md`
- **流转规则**：文档随状态变更实际移动文件（todo → doing → done）
- **产品文档流转**：`raw/` 中经用户审查确认的需求 → 移入 `deliverable/`。`context/` 中的文档仅供背景参考，不能直接进入 `deliverable/`。所有需要进入开发流程的需求必须首先在 `raw/` 中生成文档供用户审查。

## 三、新功能开发流程

### 步骤 1：需求收集

- 若需求清晰 → 在 `docs/product/raw/` 生成需求文档
- 若需求不清晰 → 调用 `/brainstorming` 与用户交互，收集清晰后 → 在 `docs/product/raw/` 生成需求文档
- 提交需求文档供用户审查

### 步骤 2：需求确认

- 用户审查并确认 → 需求文档移至 `docs/product/deliverable/`
- 用户要求修改 → 更新 `raw/` 文档，重新提交审查

### 步骤 3：开发计划

- 从主干拉取功能分支 `feature/<topic>`
- 在 `docs/development/todo/` 生成开发计划文档
- 提交开发计划供用户审查

### 步骤 4：用户审批门禁

- 用户明确同意 → 计划文档移至 `docs/development/doing/`
- 用户要求修改 → 更新计划，重新提交审查
- ⚠ 严禁在未获用户审批前编写任何实现代码

### 步骤 5：编码实现

- 按开发计划逐步实现
- 涉及前端 UI → 调用 `/frontend-design` 指导实现质量

### 步骤 6：代码审查

- 调用 `/code-review` 进行审查
- 审查通过 → 继续
- 审查未通过 → 修复后重新审查

### 步骤 7：构建验证

- 运行项目构建命令（如 `bun run build`）
- 构建成功 → 继续
- 构建失败 → 修复后重新验证

### 步骤 8：安全审查提示

- 代码审查通过后，显式提示用户是否需要进行代码风格和安全性审查
- 用户确认需要 → 完成审查后继续
- 用户跳过 → 直接进入步骤 9

### 步骤 9：完成确认

- 用户确认需求完成
- 开发文档移至 `docs/development/done/`
- 开发进展持久化到项目文档

## 四、Debug 流程

### 步骤 1：了解现状

- 阅读 `docs/product/deliverable/` 中的相关需求文档
- 阅读 `docs/development/doing/` 或 `done/` 中的相关开发文档
- 系统了解项目当前状态

### 步骤 2：系统调试

- 调用 `/systematic-debugging` 进行系统细致的调试工作
- 定位问题根因

### 步骤 3：Debug 计划

- 从主干拉取修复分支 `bugfix/<topic>`
- 在 `docs/debug/todo/` 生成 debug 计划文档（含根因分析、修复方案）
- 提交 debug 计划供用户审查

### 步骤 4：用户审批门禁

- 用户明确同意 → 计划文档移至 `docs/debug/doing/`
- 用户要求修改 → 更新计划，重新提交审查
- ⚠ 严禁在未获用户审批前编写任何实现代码

### 步骤 5：编码修复

- 按 debug 计划逐步实施修复

### 步骤 6：代码审查

- 调用 `/code-review` 进行审查
- 审查通过 → 继续
- 审查未通过 → 修复后重新审查

### 步骤 7：构建验证

- 运行项目构建命令
- 构建成功 → 继续
- 构建失败 → 修复后重新验证

### 步骤 8：安全审查提示

- 代码审查通过后，显式提示用户是否需要进行代码风格和安全性审查
- 用户确认需要 → 完成审查后继续
- 用户跳过 → 直接进入步骤 9

### 步骤 9：完成确认

- 用户确认问题已解决
- debug 文档移至 `docs/debug/done/`
- debug 进展持久化到项目文档
